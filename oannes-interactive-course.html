What is Oannes?

A tool that turns team data into insights about how fast you deliver

The Problem

Imagine you're leading a team. You want to know: How fast do we actually ship? Not "we work really hard" — actual data.

You could ask team members, but they'll guess. You could build a spreadsheet, but spreadsheets break. You could use your issue tracker's built-in reports, but they don't show the patterns that actually predict delivery.

That's where Oannes comes in. It takes your existing issue data (from Jira, Trello, Azure DevOps, GitLab, or CSV) and calculates throughput, cycle time, and other metrics that let you predict the future.

The Moment of Magic

Let's say you're in a standup and someone asks: "When will feature X ship?"

Instead of guessing, you open Oannes. You see:

  • Your team completes 8 items per week on average
  • The backlog has 40 items
  • Based on historical patterns, there's a 70% chance you'll ship everything in 5 weeks

That's a data-driven answer. That's Oannes.

Key Insight: The Separation

Notice how each component has ONE job. The connector talks to Jira. The engine calculates. The database stores. This is called separation of concerns. If Jira changes their API, you only fix the connector, not the whole system.

The Architecture in One Picture

🖥️
Frontend (React)

What you see: buttons, charts, forms. Runs in your browser. Built with React, TypeScript, Tailwind CSS. Shows dashboards, project wizard, settings.

⚙️
Backend API (FastAPI)

The server that does the thinking. Handles project creation, syncing, calculations. Built in Python, runs on your machine (or Docker).

🔗
Connectors (Multiple)

Bridges to external platforms. Each connector speaks one language: Jira, Trello, Azure DevOps, GitLab, CSV. Translates "Jira's Done = our Done stage."

📁
Database (SQLite)

A single file that stores everything: projects, work items, credentials (encrypted), metrics. No server needed — it's local and fast.

📈
Flow Engine (Python)

The math brain. Calculates throughput, cycle time, percentiles, Monte Carlo forecasts. Lives in the backend, runs after sync completes.

Where Does Your Data Go?

Here's the honest truth: Your data never leaves your machine.

Everything — projects, work items, metrics, encrypted credentials — is stored in a SQLite database file in `~/.oannes`. If you run Oannes in Docker, that's a volume you control. You own it. No cloud sync. No telemetry. No vendor lock-in.

This is by design. You're analyzing sensitive team data (how fast your team really works, who's bottlenecking work). That should never touch the internet.

Meet the Connectors

How Oannes speaks every platform's language

The Problem: Too Many Platforms

Different teams use different platforms. One team uses Jira. Another uses Trello. Another uses Azure DevOps. And they all store work items differently.

Jira has Status fields. Trello has Columns. Azure DevOps has States. Same concept, different names.

A connector is a translator. It says: "I know how to talk to Jira. When I see a Jira issue with status = 'Done', that means this item is in the done stage."

The Connector Pattern (All Connectors Work This Way)

CODE (Python) 
class BaseConnector:
    def fetch_items(self):
        pass
    
    def map_to_stage(self, status):
        pass
PLAIN ENGLISH

We define a base template that all connectors inherit from...

Every connector must implement "fetch items" — get data from the external service...

Every connector must implement "map to stage" — translate platform statuses into our standard stages (queue/in-flight/review/done)...

Once you understand this pattern, you understand all connectors:

📥
Fetch
Call the platform's API and get all work items
🗺️
Map
Translate platform-specific statuses to stages
✔️
Normalize
Extract standard fields: created, started, done dates
💾
Store
Save to database in a standard format

Real Example: How Jira Connector Works

When you create a Jira project in Oannes, you tell it: "When you see status = 'To Do', that's a queue item. When you see 'In Progress', that's in-flight. When you see 'Done', mark it done."

Then the Jira connector fetches your issues. For each issue, it:

1
Gets the created date

When the issue was first created in Jira

2
Finds the "started" moment

When status first changed from "To Do" to something else (= work started)

3
Finds the "done" moment

When status became "Done" (= work completed)

4
Saves to database

Stores all these dates plus the current stage

💡
Why This Matters for Predictions

The flow engine needs these three dates (created, started, done) to calculate lead time and cycle time. A connector's job is to extract them accurately from each platform's messy, inconsistent data. Good connectors = good data = accurate forecasts.

What Happens When a New Platform Arrives?

A team says: "We want to use Linear instead of Jira." What does the team building Oannes do?

They write a Linear connector — a new class that implements the same pattern. Fetch items from Linear API. Map Linear's "Backlog/In Progress/Done" to our stages. Extract dates. Save to the same database format.

The rest of the system doesn't change. The frontend doesn't need to know about Linear. The flow engine doesn't care. The database schema is the same.

This is why the connector pattern is so powerful — you can add platforms without touching the rest of the codebase.

The Flow Metrics Engine

How numbers become predictions — the Troy Magennis methodology

The Magic Ingredient: Historical Data

Oannes doesn't guess. It doesn't use best practices or industry benchmarks. It uses YOUR data.

Once your issues are synced, the flow engine has access to hundreds (or thousands) of completed items with real dates. This is a treasure trove for prediction.

The engine calculates eight metrics based on Troy Magennis's flow methodology:

The Eight Metrics (One at a Time)

📊
Throughput
Items completed per week, with trend line showing if you're speeding up or slowing down
⏱️
Cycle Time
Days from work start to done (50th, 85th, 95th percentile)
📅
Lead Time
Days from created to done (includes waiting in queue)
📈
WIP (Work in Progress)
How many items actively being worked on right now
🌊
CFD (Cumulative Flow)
Stacked chart showing items in each stage over time — reveals bottlenecks
⚠️
Aging WIP
Items stuck in progress longer than normal — red flags
⚙️
Flow Efficiency
Ratio of active work time to total elapsed time
🎲
Monte Carlo
Probabilistic forecast: 70% confident you'll ship X items in Y weeks

Example: Calculating Cycle Time Percentiles

Let's say your team completed 100 items in the past month. Each item has a cycle time (days from start to done). The engine sorts them:

[2, 3, 3, 4, 4, 5, 5, 5, 6, 7, ... 22, 28, 31, 45]

CODE (Python) 
import numpy as np

cycle_times = [2, 3, 3, 4, 4, 5, ...]
p50 = np.percentile(cycle_times, 50)
p85 = np.percentile(cycle_times, 85)
p95 = np.percentile(cycle_times, 95)
PLAIN ENGLISH

Import NumPy, a math library...

Give it the cycle times from all completed items...

Calculate the 50th percentile (median — half the items are faster, half slower)...

Calculate 85th percentile (this is what you should plan for — realistic)...

Calculate 95th percentile (worst case — use for risk planning)...

So if p50 = 5 days, p85 = 12 days, p95 = 25 days, you know:

  • Half your work gets done in 5 days or less
  • Most work (85% historically) is done in 12 days or less — use this for planning
  • Only the hardest 5% takes longer than 25 days

The Bottleneck Detective: CFD

The Cumulative Flow Diagram is a stacked area chart showing how many items are in each stage (queue, in-flight, review, done) over time.

If you see a flat line in the in-flight area while done items keep growing, that's great — items are moving through. But if the in-flight area gets thick and stops growing, items are stuck. That's your bottleneck.

💡
The CFD Pattern

In a healthy flow, the CFD looks like smooth diagonal lines stacking up. Queue grows, then items move to in-flight, then to done. If you see a horizontal band that's not growing, work is stuck there. Time to investigate.

Making Dashboards Smart: Caching

Every time the user switches tabs, metrics shouldn't refetch. That's wasteful. TanStack Query caches the data.

First visit to dashboard: fetch from server, render chart, cache it.

Second visit (within 5 minutes): show cached version instantly. In the background, silently refetch fresh data. When it arrives, update the chart without interrupting the user.

This is called stale-while-revalidate.

Predicting the Future

Monte Carlo simulation — the most powerful metric Oannes offers

The Question Nobody Can Answer

"When will we ship 50 items?"

It's a fair question. You have 50 items in the backlog. Your team completes 8 items per week. Math says: 50 ÷ 8 = 6.25 weeks, right?

Wrong. That assumes perfect consistency. Real teams are messier. Some weeks you do 6, some weeks you do 10. Some items are hard, some are easy.

Monte Carlo simulation handles that messiness. It says: "Based on your historical variability, here are the probabilities."

How Monte Carlo Works (Simplified)

Imagine a lottery machine with balls labeled with your past weekly throughputs: [5, 6, 6, 7, 8, 8, 8, 9, 10, 12].

The simulation runs 10,000 times:

1
Draw a random ball

Pick one of your historical weekly throughputs at random

2
Add it to the total

You've completed that many items this week

3
Repeat until you hit 50

Keep drawing and summing until your total reaches 50

4
Count how many weeks it took

This is one simulated scenario

Do this 10,000 times. Sometimes it took 5 weeks, sometimes 7, sometimes 8. When you plot all 10,000 results, you get a distribution:

  • 5% chance: 4 weeks (you got lucky)
  • 25% chance: 5 weeks
  • 50% chance: 6 weeks (median)
  • 85% chance: 7 weeks (use this for planning)
  • 95% chance: 9 weeks (risk planning)
💡
Why 85%?

In project management, the 85th percentile is the sweet spot. It's ambitious but achievable. You're accounting for the variability in your real work, not hoping for perfection.

How Oannes Implements Monte Carlo

CODE (Python) 
def monte_carlo(throughput_history, target):
    results = []
    for _ in range(10000):
        total = 0
        weeks = 0
        while total < target:
            weekly = np.random.choice(
              throughput_history)
            total += weekly
            weeks += 1
        results.append(weeks)
    return results
PLAIN ENGLISH

Define a function that takes historical throughput and a target (50 items)...

Create an empty list to store results...

Run 10,000 simulations...

For each simulation, start with 0 items completed...

Keep adding random weeks of throughput...

When you reach the target, record how many weeks it took...

Return all 10,000 results...

The beauty of this approach is it uses your actual data. Your team's actual variability. Your actual patterns. Not someone's formula from a management book.

The Real Power: Risk Conversations

Oannes gives you language for tough conversations:

PM: "When can we ship?"

You: "Oannes says 85% chance in 7 weeks. That's based on our actual 3-month history, not guessing."

PM: "What if we need it in 5 weeks?"

You: "That's the 25th percentile. 75% chance we miss it. We could hire someone, remove some scope, or accept the risk."

No emotion. Just probabilities. That's powerful.

How to Steer AI (When Building Tools Like This)

Now that you understand how Oannes works, here's how you'd tell an AI assistant to improve it:

1
Name the component, not the symptom

Instead of "the forecasting is wrong," say "The Monte Carlo simulation needs vectorized NumPy calculations to handle large datasets faster."

2
Respect separation of concerns

Instead of "add forecasting to the frontend," say "Add a new API endpoint for forecasts, then the frontend can display them."

3
Propose patterns, not hacks

Instead of "let's use caching somehow," say "Implement stale-while-revalidate caching with TanStack Query."

4
Think about data flow

Where does data enter the system? How does it transform? Where does it exit? Draw it if you have to.
