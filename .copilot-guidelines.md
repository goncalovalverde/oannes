## Enforced Discipline

### TDD is Non-Negotiable

Every change (feature, fix, refactor) follows:
- **RED**: Write failing tests first
- **GREEN**: Minimal implementation to pass tests
- **REFACTOR**: Clean code while tests pass
- **VERIFY**: Full suite passes with ≥80% coverage

### Required Reading

Before starting work, read:
- [ ] `ARCHITECTURE.md` — Understand system design, metrics, connectors, key decisions
- [ ] `CONTRIBUTING.md` — Understand TDD workflow, test patterns, quality standards
- [ ] Relevant sections of `README.md` — For overview

### Coverage Enforcement

- Backend: ≥80% (enforced in CI via `pytest.ini`)
- Frontend: ≥70%
- No exceptions for "simple" changes

## Test Quality Standards (Non-Negotiable)

✅ **DO**:
- Write tests BEFORE implementation
- Use fixtures for setup (db, client, sample_project)
- Test edge cases (empty, null, invalid, error paths)
- Make test names describe the scenario clearly
- Use deterministic seeds for randomness

❌ **DON'T**:
- Use `time.sleep()` in tests (flaky!)
- Call real external APIs (mock them)
- Skip or mark tests as pending (fix root cause)
- Couple tests to implementation details
- Create global mutable state between tests
